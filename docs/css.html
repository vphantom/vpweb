Vanilla+ CSS Framework

Introduction

There are already plenty of CSS frameworks, with varying levels of complexity. This one attempts to be minuscule, restricted to pure CSS, without sacrificing esthetics which contribute to usability. It strays from the extreme simplicity of the flat layered "paper" trend, in favor of more classic visual cues enhanced to use modern browser capabilities.

Almost all classes begin with the prefix vp- to prevent collisions with other frameworks. For brevity, there is no prefix for attribute classes like is-something, has-something and size-something. Truly independent features like messages have their own custom HTML tag names, like vp-message instead of being set up as a class to be applied on a div or a span

Compiled CSS files are located in dist/. You may use vpweb.css to get the entire framework at once, or just some of the individual files described below. Be sure to import core.css first in order to avoid priority side-effects.

Layout layout.css

Responsive

Certain classes restrict which devices can see an element, by viewport width or display type. Note that the names "mobile", "tablet" and "desktop" are used as nicknames for the three responsive size ranges, not as true device identification. CSS media queries also use a "device pixel ratio" to define their own meaning for "px", so those are not actual pixels.

Mobile:
Up to 767px, .vp-mobile-only, .vp-tablet-down
Tablet:
From 768px to 991px, .vp-tablet-only, .vp-tablet-up, .vp-tablet-down
Desktop:
From 992px up, .vp-tablet-up, .vp-desktop-only
Print:
.vp-print-only, .vp-screen-only (non-screen is considered "print")

Block .vp-block

Makes any element (including buttons) a block and full width.

Grid grid.css

Two bases (10 and 24) were chosen for their versatility: areas can be divided by 2, 3, 4, 5, 6, 8, 10, 12 and 24.

Rows are .vp-row-10 or .vp-row-24 according to whether you want the row to be divided in 10ths or 24ths. Columns are .vp-col-X where X is the multiple width. Rows wrap, so you only need to use multiple row blocks for splitting incomplete lines.

An <img> directly inside a .vp-col-* or wrapped in just an a automatically gets full width.

Caution: while .vp-col-* spacing is sensitive to .vp-density-* given to its parent row or that row's parent, it also causes a small negative margin on the left and right ends of rows. Therefore, using a page-wide row with non-seamless density will cause horizontal scrolling. As long as a row's container has at least as much spacing, it is not an issue. (i.e. A seamless or spaced row fits inside a page-wide relaxed or spaced container.)

Responsive

.vp-doubling .vp-col-*
Makes grid cells double in relative width when stepping down from "desktop" to "tablet" size, and double again down to "mobile" size. (i.e. A 1/12th cell would become 1/6th wide in "tablet" viewports and 1/3rd wide in "mobile" viewports.)
.vp-stacking .vp-col-*
Makes grid cells go full-width for "tablet" and "mobile" sizes.

Forms forms.css

Note that buttons and input widgets with :focus get a z-index of 1000 or 1001 in addition to the halo. Note also that margins between some elements is affected by form.vp-density-*.

Fields label.vp-field .vp-label + input|select|textarea

In addition to text areas, typical input types compatible with this class are: email, number, password, tel, text, url and possibly search. Fields expand to fill their container's width, which is handy in grids. We implement infield top aligned labels for their clarity.

Icon Buttons .vp-icon-button

Unlike other form styles, icon buttons have fixed sizes based on touch display ability and their priority. Normal by default, they may have .is-secondary or .is-priority. On touch devices, these three sizes are 11mm, 15mm and 19mm.

Button groups .vp-group button, .vp-group .vp-button

Theming

Normal buttons automatically adapt to any background/foreground colors and font sizes set on their elements, as well as their parent background color. Icon buttons adapt to font size and their parent background color.

Tooltips tooltips.scss

For uniformity with touch devices, our tooltips are touch/click activated. This requires adding the ontouchstart attribute to body and adding tabindex="0" to non-clickable elements like span.

To evade their trigger or parent element's width, tooltips don't break on white space. Keep this in mind for placement to avoid overflowing the viewport. For element tooltips, you may insert manual breaks as desired.

Attribute [vp-tooltip]

To give any element (except inputs) a clickable tooltip, the plain text content of the tooltip need simply be supplied in a vp-tooltip="" attribute. Optional accompanying boolean attributes vp-tooltip-below and vp-tooltip-right affect alignment and position.

Click this sentence for example.

Element vp-tooltip

To deal with inputs and to use inline HTML in a tooltip, use vp-tooltip which must immediately follow a focusable element such as input, and both need to be inside a position:relative block. For our forms, this is easily achieved with a field (an input inside its label). Optional boolean attributes _below and _right affect alignment and position.

Transitions transitions.css

To guarantee that no browsers "flash" transitions on load, some rules depend on a parent having the "loaded" class. To use them, add the loaded class to body after the first paint is likely over. In some browsers, a simple "onload" is insufficient, thus:

<body onload="setTimeout(function(){document.body.classList.add('loaded')})">
